Plano estratégico — Bot ADNY

Assistente no WhatsApp (um grupo) · somente leitura · APIs gratuitas quando possível · Documento único (HTML)

Objetivos do projeto

Objetivo principal Permitir que integrantes marquem o bot (ex.: @ADNY) em um grupo específico e recebam respostas em texto com dados do sistema de pedidos (planilha via Google Apps Script), por exemplo: lista de pedidos em aberto com resumo.

Restrições O bot não cria, edita, remove nem anexa dados no sistema — apenas consulta e formata a resposta.

Isolamento Código e documentação do bot ficam na pasta BOT-ADNY/, sem misturar com o front principal (index.html, script.js, etc.).

Fora do escopo inicial Alterar pedidos, imagens no Drive, status na planilha; pastas ROANTONE, ListaPreencher, CalculadoraMalha (salvo decisão futura).

PC que será usado (host de produção — decidido)

Perfil: PC dedicado apenas ao bot + Windows 10 + tarefas padrão (antivírus, atualizações). Sem Photoshop, Corel, Chrome pesado no uso diário.

  • CPU: Intel Core i7 (6ª geração)
  • RAM: 12 GB DDR4
  • GPU: NVIDIA GTX 960 4 GB (Ollama pode ficar em CPU se GPU não for estável)
  • Armazenamento: SSD 250 GB
  • Sistema: Windows 10

Desenvolvimento e testes: podem ocorrer no PC de trabalho atual; produção no i7 dedicado acima.

Nota: Fase 1 (regras + consultas) é a mais adequada a este hardware; Ollama (Fase 2) usa modelos 1B–3B quantizados, tipicamente em CPU.

Softwares utilizados

  • Windows 10 — sistema base do host
  • Node.js (LTS) ou Python 3.x — runtime do servidor webhook do bot (escolher um na implementação)
  • Git — opcional (atualização via repositório)
  • Cloudflare Tunnel ou ngrok (free tier) ou equivalente — HTTPS para webhook quando não houver IP/certificado próprio
  • Ollama — opcional (Fase 2, LLM local)
  • NSSM ou Agendador de Tarefas — opcional — manter o bot rodando / iniciar com o Windows
  • Firewall do Windows — regras se necessário expor porta local (túnel costuma evitar)

APIs e serviços (nuvem)

  • WhatsApp Cloud API (Meta) — recebimento/envio no número Business no grupo autorizado
  • Web App do Google Apps Script — URL .../exec do sistema de pedidos, uso apenas com ações de leitura: listarPedidos, buscarPedido, buscarPedidos, relatorioPedidos, obterConfigCalculo (se necessário), etc.
  • Google Gemini API — opcional, Fase 2, free tier (orquestrador)
  • Groq API — opcional, Fase 2, se houver tier gratuito e for escolhido

Cotas gratuitas Meta/Google devem ser validadas na conta no momento do deploy. Arquitetura base não exige SaaS pago obrigatório.

Fluxo da mensagem (lógica)

  1. WhatsApp envia evento ao webhook (HTTPS).
  2. Validar grupo permitido, menção ao bot e assinatura (quando aplicável).
  3. Interpretar texto (Fase 1: regras; Fase 2: opcional LLM com tools fechadas).
  4. Consultar Apps Script apenas com leitura; montar texto; fragmentar se exceder limite de caracteres.
  5. Enviar resposta ao grupo via Cloud API.

Roteiro — Passos para implementação e marcos

  1. Conta e POC WhatsApp
    Criar/configurar app na Meta (WhatsApp Cloud API), número Business no grupo alvo, webhook apontando para URL pública (túnel em dev). Objetivo: provar leitura de mensagens do grupo e envio de resposta.
  2. Estrutura BOT-ADNY
    Servidor HTTP(S) com rota de webhook, variáveis em .env, logs. Objetivo: receber payload da Meta e responder 200 OK com verificação de assinatura.
  3. Adaptador somente leitura (Apps Script)
    Implementar chamadas GET a listarPedidos, buscarPedido, buscarPedidos, relatorioPedidos conforme intenção mapeada. Objetivo: JSON → texto legível; truncar/quebrar mensagens longas.
  4. Fase 1 — Motor de regras
    Palavras-chave e comandos fixos (ex.: pedidos abertos, busca por ID ou 4 dígitos). Objetivo: cobrir 80% do uso sem IA.
  5. Segurança mínima
    Allowlist de um group_id; rate limit; nunca chamar POST de escrita. Objetivo: reduzir abuso e vazamento de URL da planilha.
  6. Endurecimento backend (recomendado)
    Token ou segunda implantação read-only no Apps Script. Objetivo: mesmo com URL vazada, escrita bloqueada para o canal do bot.
  7. Deploy no PC i7
    Copiar projeto, configurar .env, túnel estável, iniciar serviço com NSSM ou Tarefa Agendada; energia: suspensão desligada se precisar 24/7. Objetivo: disponibilidade estável.
  8. Fase 2 (opcional)
    Ollama (modelo pequeno, CPU) ou Gemini/Groq free tier como orquestrador com tools validadas. Objetivo: linguagem mais livre sem abrir acesso a escrita.
  9. Empacotamento futuro
    Documentar iniciar-bot.bat, backup de .env, e opcional .exe (pkg/PyInstaller). Objetivo: reinstalar rápido em outro hardware se necessário.
